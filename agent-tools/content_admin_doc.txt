# تفاصيل الميزات

## F01 — التسجيل واعتماد الحسابات والدخول

**الهدف:** يضمن الأدمن أن كل حساب يدخل المنصة موثّق ومعتمد: مراجعة طلبات المطاعم والسائقين قبل تفعيلها، وإدارة حسابات العملاء (تفعيل/تعطيل)، بحيث لا يظهر أي مطعم أو يعمل أي سائق قبل موافقته الصريحة.

**مصدر الشاشات:** مستنتجة من خطوات الـ Workflow

### فلو الشاشات

```mermaid
flowchart TD
  S1["لوحة المتطلبات / المدخلات"]
  S2["طلبات التسجيل المعلّقة"]
  S3["يفتح طلب المطعم"]
  S4["اعتماد"]
  S5["يفتح طلب السائق العام"]
  S6["يدير حسابات العملاء"]
  S7["نسيت كلمة المرور"]
  S1 --> S2
  S2 --> S3
  S3 --> S4
  S4 --> S5
  S5 --> S6
  S6 --> S7
```

### تفاصيل ومحتويات كل لوحة

#### لوحة المتطلبات / المدخلات _(مستنتجة)_

**محتويات:**

- لوحة تحكم ويب (Angular) بصلاحيات الأدمن داخل نطاق دولته.
- طلب تسجيل مطعم يتضمّن بيانات الشركة الرسمية + الوثائق المرفوعة (راجع F22).
- طلب تسجيل سائق عام يتضمّن البيانات الشخصية والرخصة وبيانات المركبة (راجع F23).
- قائمة العملاء المسجّلين عبر التطبيق (Email/Phone + كلمة المرور + المنطقة + الموافقة على الشروط).

#### طلبات التسجيل المعلّقة _(مستنتجة)_

**محتويات:**

- يفتح الأدمن قائمة  مقسّمة إلى: مطاعم، سائقون عامّون

#### يفتح طلب المطعم _(مستنتجة)_

**محتويات:**

- يفتح طلب المطعم: يراجع البيانات الرسمية والوثائق، ويتحقق من اكتمالها وصلاحية تواريخها

#### اعتماد _(مستنتجة)_

**محتويات:**

- يتخذ قرارًا:  (تفعيل الحساب وإظهاره للعملاء) أو رفض/طلب استكمال مع ذكر السبب

#### يفتح طلب السائق العام _(مستنتجة)_

**محتويات:**

- يفتح طلب السائق العام: يتحقق من الرخصة ووثيقة المركبة ثم يمنح الموافقة النهائية للتفعيل

#### يدير حسابات العملاء _(مستنتجة)_

**محتويات:**

- يدير حسابات العملاء: بحث، تفعيل، أو تعطيل حساب عند الحاجة (إساءة استخدام/طلب العميل)

#### نسيت كلمة المرور _(مستنتجة)_

**محتويات:**

- يتابع سجل الدخول والصلاحيات، ويعالج طلبات  الاستثنائية إن لزم

### مسار المستخدم (خطوات)

1. يفتح الأدمن قائمة «طلبات التسجيل المعلّقة» مقسّمة إلى: مطاعم، سائقون عامّون.
2. يفتح طلب المطعم: يراجع البيانات الرسمية والوثائق، ويتحقق من اكتمالها وصلاحية تواريخها.
3. يتخذ قرارًا: **اعتماد** (تفعيل الحساب وإظهاره للعملاء) أو **رفض/طلب استكمال** مع ذكر السبب.
4. يفتح طلب السائق العام: يتحقق من الرخصة ووثيقة المركبة ثم يمنح الموافقة النهائية للتفعيل.
5. يدير حسابات العملاء: بحث، تفعيل، أو تعطيل حساب عند الحاجة (إساءة استخدام/طلب العميل).
6. يتابع سجل الدخول والصلاحيات، ويعالج طلبات «نسيت كلمة المرور» الاستثنائية إن لزم.

### الحالات والاستثناءات

- وثائق ناقصة أو منتهية → يبقى الحساب «قيد المراجعة» ولا يُفعّل، مع إشعار صاحب الطلب بالسبب.
- محاولة مطعم/سائق العمل قبل الاعتماد → ممنوع نظاميًا (الحساب غير ظاهر/غير قابل للإسناد).
- تعطيل حساب عميل له اشتراك نشط → يجب معالجة الاشتراك (إلغاء/تجميد) قبل أو مع التعطيل.
- Social Login للعميل مدعوم لاحقًا ولا يغيّر منطق الاعتماد للمطاعم والسائقين.

## F02 — إدارة نموذج الاشتراك (المدد/البرامج/الباقات)

**الهدف:** يتحكم الأدمن ديناميكيًا في العمود الفقري لنموذج الاشتراك: مدد الاشتراك، البرامج الغذائية، والباقات، بحيث يضيف/يعدّل/يلغي أي عنصر من لوحة التحكم دون الحاجة لتغيير برمجي.

**مصدر الشاشات:** مستنتجة من خطوات الـ Workflow

### فلو الشاشات

```mermaid
flowchart TD
  S1["لوحة المتطلبات / المدخلات"]
  S2["نموذج الاشتراك"]
  S3["المدد:"]
  S4["البرامج الغذائية:"]
  S5["الباقات:"]
  S6["المطعم (لتسعيرها)"]
  S7["لوحة — خطوة 6"]
  S1 --> S2
  S2 --> S3
  S3 --> S4
  S4 --> S5
  S5 --> S6
  S6 --> S7
```

### تفاصيل ومحتويات كل لوحة

#### لوحة المتطلبات / المدخلات _(مستنتجة)_

**محتويات:**

- صلاحية إدارة الكتالوج في لوحة الأدمن.
- تعريف مدد الاشتراك: شهري (26 يوم عمل بدون الجمعة)، أسبوعين (12 يوم)، أسبوع (6 أيام)، مخصص (من يوم واحد فأكثر).
- قائمة البرامج الغذائية المطلوبة: نزول وزن، ضخامة عضلية، محافظة، كيتو.
- قائمة الباقات: باقة كاملة (إفطار + وجبتان رئيسيتان + سناك + سلطة)، باقة الغداء (وجبة رئيسية + سلطة)، باقة مخصصة (مزيج يحدده الأدمن).

#### نموذج الاشتراك _(مستنتجة)_

**محتويات:**

- يفتح الأدمن قسم  في لوحة التحكم

#### المدد: _(مستنتجة)_

**محتويات:**

- يفعّل/يعطّل كل مدة ويضبط عدد أيامها ونسبة عمولتها المرتبطة (تُستخدم في F04)

#### البرامج الغذائية: _(مستنتجة)_

**محتويات:**

- يضيف برنامجًا جديدًا (اسم + وصف + لغتان) أو يعدّل/يلغي القائمة

#### الباقات: _(مستنتجة)_

**محتويات:**

- يعرّف مكوّنات كل باقة، ويُنشئ الباقة المخصصة كمزيج من العناصر

#### المطعم (لتسعيرها) _(مستنتجة)_

**محتويات:**

- يحفظ ويعتمد النشر
- تنعكس الخيارات فورًا في تطبيق العميل وفي لوحة المطعم (لتسعيرها)

#### لوحة — خطوة 6 _(مستنتجة)_

**محتويات:**

- يتابع أن كل برنامج/باقة مرتبط بأسعار مدخلة من المطاعم لـ26 يوم قبل إتاحته للاشتراك

### مسار المستخدم (خطوات)

1. يفتح الأدمن قسم «نموذج الاشتراك» في لوحة التحكم.
2. **المدد:** يفعّل/يعطّل كل مدة ويضبط عدد أيامها ونسبة عمولتها المرتبطة (تُستخدم في F04).
3. **البرامج الغذائية:** يضيف برنامجًا جديدًا (اسم + وصف + لغتان) أو يعدّل/يلغي القائمة.
4. **الباقات:** يعرّف مكوّنات كل باقة، ويُنشئ الباقة المخصصة كمزيج من العناصر.
5. يحفظ ويعتمد النشر؛ تنعكس الخيارات فورًا في تطبيق العميل وفي لوحة المطعم (لتسعيرها).
6. يتابع أن كل برنامج/باقة مرتبط بأسعار مدخلة من المطاعم لـ26 يوم قبل إتاحته للاشتراك.

### الحالات والاستثناءات

- إلغاء برنامج/باقة عليها اشتراكات نشطة → يجب الاحتفاظ بها للمشتركين الحاليين وإخفاؤها عن الجدد فقط.
- برنامج بلا أسعار مدخلة من أي مطعم → لا يُتاح للاختيار حتى تتوفر أسعار.
- تعديل مكوّنات باقة لا يؤثر على الاشتراكات الجارية ضمن نافذة 48 ساعة.

## F03 — إدارة تصنيف المطاعم (Basic/Platinum/Elite)

**الهدف:** يضبط الأدمن الحدود السعرية للتصنيفات الثلاثة، فيصنّف النظام كل مطعم **تلقائيًا** حسب سعر اشتراكه لـ26 يومًا. التصنيف داخلي بين الأدمن وشركات المطاعم ويحدد جودة/سعر البوكس الذي يراه العميل.

**مصدر الشاشات:** مستنتجة من خطوات الـ Workflow

### فلو الشاشات

```mermaid
flowchart TD
  S1["لوحة المتطلبات / المدخلات"]
  S2["التصنيفات"]
  S3["لوحة — خطوة 2"]
  S4["لوحة — خطوة 3"]
  S5["لوحة — خطوة 4"]
  S6["لوحة — خطوة 5"]
  S1 --> S2
  S2 --> S3
  S3 --> S4
  S4 --> S5
  S5 --> S6
```

### تفاصيل ومحتويات كل لوحة

#### لوحة المتطلبات / المدخلات _(مستنتجة)_

**محتويات:**

- سعر اشتراك كل مطعم لمدة 26 يومًا (مدخل من المطعم لكل برنامج/باقة).
- الحدود السعرية التي يضعها الأدمن لكل تصنيف.
- مثال للحدود (ديناميكية): Basic < 100 د.ك | Platinum < 150 د.ك | Elite < 200 د.ك.

#### التصنيفات _(مستنتجة)_

**محتويات:**

- يفتح الأدمن قسم  ويحدد الحد السعري الأعلى لكل تصنيف

#### لوحة — خطوة 2 _(مستنتجة)_

**محتويات:**

- عند إدخال/تغيير سعر مطعم، يحسب النظام تلقائيًا التصنيف بمقارنة سعر 26 يومًا بالحدود

#### لوحة — خطوة 3 _(مستنتجة)_

**محتويات:**

- يراجع الأدمن جدول المطاعم وتصنيفها الناتج ويتأكد من منطقية التوزيع

#### لوحة — خطوة 4 _(مستنتجة)_

**محتويات:**

- يعدّل الحدود عند الحاجة → يعيد النظام تصنيف كل المطاعم المتأثرة فورًا

#### لوحة — خطوة 5 _(مستنتجة)_

**محتويات:**

- يتحقق أن قيمة البوكس اليومي لكل مطعم = سعر اشتراك المطعم (26 يوم) ÷ 26

### مسار المستخدم (خطوات)

1. يفتح الأدمن قسم «التصنيفات» ويحدد الحد السعري الأعلى لكل تصنيف.
2. عند إدخال/تغيير سعر مطعم، يحسب النظام تلقائيًا التصنيف بمقارنة سعر 26 يومًا بالحدود.
3. يراجع الأدمن جدول المطاعم وتصنيفها الناتج ويتأكد من منطقية التوزيع.
4. يعدّل الحدود عند الحاجة → يعيد النظام تصنيف كل المطاعم المتأثرة فورًا.
5. يتحقق أن قيمة البوكس اليومي لكل مطعم = سعر اشتراك المطعم (26 يوم) ÷ 26.

### الحالات والاستثناءات

- مطعم سعره أعلى من حد Elite → لا يدخل أي تصنيف حتى يضبط الأدمن الحدود أو يراجع السعر.
- تغيير الحدود يعيد تصنيف المطاعم لكنه لا يكسر اشتراكات نشطة ضمن نافذة 48 ساعة.
- مطعم بلا سعر مدخل → لا يُصنّف ولا يظهر للعملاء.

## F04 — إدارة التسعير والعمولات الديناميكية

**الهدف:** يتحكم الأدمن في كل نسب وحدود التسعير: عمولة المدة (التي تُحمّل على العميل)، والعمولة الديناميكية لكل مطعم (التي تُخصم من سعر البوكس المتفق عليه)، بينما يحدّث النظام المتوسطات تلقائيًا.

**مصدر الشاشات:** مستنتجة من خطوات الـ Workflow

### فلو الشاشات

```mermaid
flowchart TD
  S1["لوحة المتطلبات / المدخلات"]
  S2["التحكم"]
  S3["لوحة — خطوة 2"]
  S4["متوسط سعر القروب"]
  S5["متوسط البوكس اليومي"]
  S6["سعر اشتراك العميل"]
  S7["لوحة — خطوة 6"]
  S1 --> S2
  S2 --> S3
  S3 --> S4
  S4 --> S5
  S5 --> S6
  S6 --> S7
```

### تفاصيل ومحتويات كل لوحة

#### لوحة المتطلبات / المدخلات _(مستنتجة)_

**محتويات:**

- أسعار المطاعم لـ26 يومًا داخل كل تصنيف (لحساب متوسط القروب).
- نسب العمولة للمدة (ديناميكية): شهر 20%، أسبوعين 25%، أسبوع 30%، مخصص 25%.
- نسبة عمولة ديناميكية لكل مطعم (مثال: مطعم A 15%، مطعم B 20%).

#### التحكم _(مستنتجة)_

**محتويات:**

- يضبط الأدمن نسبة العمولة لكل مدة اشتراك من لوحة التحكم

#### لوحة — خطوة 2 _(مستنتجة)_

**محتويات:**

- يحدد نسبة العمولة الديناميكية لكل مطعم على حدة

#### متوسط سعر القروب _(مستنتجة)_

**محتويات:**

- يحسب النظام تلقائيًا  لكل تصنيف (مع الالتزام بالبرنامج والباقة)

#### متوسط البوكس اليومي _(مستنتجة)_

**محتويات:**

- يشتق  = متوسط القروب لـ26 يوم ÷ 26

#### سعر اشتراك العميل _(مستنتجة)_

**محتويات:**

- يحسب  = (متوسط البوكس اليومي × أيام الاشتراك) × (1 + نسبة عمولة المدة)

#### لوحة — خطوة 6 _(مستنتجة)_

**محتويات:**

- يتابع الأدمن إعادة حساب المتوسطات تلقائيًا عند انضمام مطاعم جديدة أو تغيير أسعارها

### مسار المستخدم (خطوات)

1. يضبط الأدمن نسبة العمولة لكل مدة اشتراك من لوحة التحكم.
2. يحدد نسبة العمولة الديناميكية لكل مطعم على حدة.
3. يحسب النظام تلقائيًا **متوسط سعر القروب** لكل تصنيف (مع الالتزام بالبرنامج والباقة).
4. يشتق **متوسط البوكس اليومي** = متوسط القروب لـ26 يوم ÷ 26.
5. يحسب **سعر اشتراك العميل** = (متوسط البوكس اليومي × أيام الاشتراك) × (1 + نسبة عمولة المدة).
6. يتابع الأدمن إعادة حساب المتوسطات تلقائيًا عند انضمام مطاعم جديدة أو تغيير أسعارها.

### الحالات والاستثناءات

- انضمام مطعم جديد أو تغيير سعره → تحديث فوري لمتوسط القروب وسعر العميل للاشتراكات الجديدة.
- الاشتراكات النشطة لا يتغير سعرها بأثر رجعي عند تعديل النسب.
- تصنيف بلا مطاعم كافية → متوسط غير ممثّل؛ ينبّه الأدمن قبل إتاحته.

## F06 — قاعدة 48 ساعة واستثناءات الأدمن

**الهدف:** يشرف الأدمن على تطبيق قاعدة 48 ساعة التي تجمّد التعديل قبل التوصيل بأقل من يومين، ويملك **وحده** صلاحية الاستثناء في الحالات الطارئة عبر واتساب لضمان عدم تضرر العميل دون كسر الانضباط التشغيلي.

**مصدر الشاشات:** مستنتجة من خطوات الـ Workflow

### فلو الشاشات


**الهدف:** يضبط الأدمن قواعد نظام النقاط والاستبدال: متى يكسب العميل نقاطًا وكم، وكيف يستبدلها، بما يحفّز الاشتراك والتجديد والتفاعل دون الإضرار بالاقتصاد المالي للمنصة.

**مصدر الشاشات:** مستنتجة من خطوات الـ Workflow

### فلو الشاشات

```mermaid
flowchart TD
  S1["لوحة المتطلبات / المدخلات"]
  S2["لوحة — خطوة 1"]
  S3["لوحة — خطوة 2"]
  S4["لوحة — خطوة 3"]
  S5["المكافآت للعميل (رصيد + طرق الكسب + سجل + استبدال)"]
  S6["لوحة — خطوة 5"]
  S7["لوحة — خطوة 6"]
  S1 --> S2
  S2 --> S3
  S3 --> S4
  S4 --> S5
  S5 --> S6
  S6 --> S7
```

### تفاصيل ومحتويات كل لوحة

#### لوحة المتطلبات / المدخلات _(مستنتجة)_

**محتويات:**

- أحداث كسب النقاط: الاشتراك، التجديد، الطلب، التقييم، الإحالة.
- جدول قيمة النقاط لكل حدث وقواعد الاستبدال والانتهاء.
- ميزانية/سقف المكافآت ضمن الخطة المالية.

#### لوحة — خطوة 1 _(مستنتجة)_

**محتويات:**

- يحدد الأدمن عدد النقاط الممنوحة لكل حدث (اشتراك/تجديد/طلب/تقييم/إحالة)

#### لوحة — خطوة 2 _(مستنتجة)_

**محتويات:**

- يضبط قواعد الاستبدال (قيمة النقطة، الحد الأدنى للاستبدال، المكافآت المتاحة)

#### لوحة — خطوة 3 _(مستنتجة)_

**محتويات:**

- يضبط صلاحية النقاط ومدة انتهائها إن وُجدت

#### المكافآت للعميل (رصيد + طرق الكسب + سجل + استبدال) _(مستنتجة)_

**محتويات:**

- ينشر القواعد فتظهر في شاشة المكافآت للعميل (رصيد + طرق الكسب + سجل + استبدال)

#### لوحة — خطوة 5 _(مستنتجة)_

**محتويات:**

- يتابع تقارير النقاط (مكتسبة/مستخدمة/منتهية) وأثرها على التجديد

#### لوحة — خطوة 6 _(مستنتجة)_

**محتويات:**

- يعدّل القواعد ديناميكيًا حسب الأداء والميزانية

### مسار المستخدم (خطوات)

1. يحدد الأدمن عدد النقاط الممنوحة لكل حدث (اشتراك/تجديد/طلب/تقييم/إحالة).
2. يضبط قواعد الاستبدال (قيمة النقطة، الحد الأدنى للاستبدال، المكافآت المتاحة).
3. يضبط صلاحية النقاط ومدة انتهائها إن وُجدت.
4. ينشر القواعد فتظهر في شاشة المكافآت للعميل (رصيد + طرق الكسب + سجل + استبدال).
5. يتابع تقارير النقاط (مكتسبة/مستخدمة/منتهية) وأثرها على التجديد.
6. يعدّل القواعد ديناميكيًا حسب الأداء والميزانية.

### الحالات والاستثناءات

- إساءة استغلال (نقاط من أحداث وهمية) → تجميد/تصحيح الرصيد وتوثيق الحالة.
- إلغاء اشتراك مُنحت عليه نقاط → مراجعة سحب النقاط غير المستحقة.
- تغيير قيمة النقطة → يطبّق على الكسب الجديد دون مصادرة المكتسب تعسفيًا.

2. ينشئ أكواد برومو مرتبطة بحملات/مؤثرين (نسبة/قيمة، تاريخ صلاحية، حد استخدام).
3. يربط كل كود/رابط بحملة لتتبع أدائها.
4. يتأكد أن البرومو كود يظهر للعميل في شاشة الدفع ويُطبَّق بشكل صحيح.
5. يتابع تقارير الاستخدام: عدد المدعوين، المكافآت، مرات استخدام الأكواد، أثرها على المبيعات.
6. يوقف/يعدّل أي كود أو حملة عند انتهائها أو سوء استخدامها.

### الحالات والاستثناءات

- استخدام كود منتهٍ/مستنفد → يرفضه النظام عند الدفع مع رسالة واضحة.
- محاولة احتيال إحالة (حسابات وهمية) → كشف وتعطيل ومراجعة المكافآت.
- تعارض كود مع عرض آخر → قواعد أولوية يحددها الأدمن.

```mermaid
  S4["لوحة — خطوة 3"]

  S3 --> S4

  S5 --> S6
  S6 --> S7
```

### تفاصيل ومحتويات كل لوحة

#### لوحة المتطلبات / المدخلات _(مستنتجة)_

**محتويات:**

- اشتراك رئيسي يديره «مدير العائلة».
- أفراد عائلة بحسابات منفصلة مرتبطة بالاشتراك الرئيسي.
- قواعد الحصص والصلاحيات (الفرد يستخدم حصته فقط ولا يعدّل الخطة).

#### يتابع الأدمن لوحات العائلات _(مستنتجة)_

**محتويات:**

- يتابع الأدمن لوحات العائلات: عدد الأفراد، حالاتهم، الطلبات المرتبطة

#### لوحة — خطوة 2 _(مستنتجة)_

**محتويات:**

- يتأكد أن مدير العائلة يضيف أفرادًا بحسابات منفصلة دون منحهم صلاحية تعديل الاشتراك الرئيسي

#### لوحة — خطوة 3 _(مستنتجة)_

**محتويات:**

- يراقب أن كل فرد يستخدم حصته فقط ولا يعدّل الخطة

#### يدعم عمليات الفصل _(مستنتجة)_

**محتويات:**

- يدعم عمليات الفصل: تأكيد → اختيار تحويل لحساب مستقل أو ترقية لاشتراك فردي → رسالة نجاح

#### لوحة — خطوة 5 _(مستنتجة)_

**محتويات:**

- يتدخل في النزاعات (حصص/فصل) عند تصعيدها له

#### لوحة — خطوة 6 _(مستنتجة)_

**محتويات:**

- يربط بيانات العائلة بالتقارير والمحاسبة عند الحاجة

### مسار المستخدم (خطوات)

1. يتابع الأدمن لوحات العائلات: عدد الأفراد، حالاتهم، الطلبات المرتبطة.
2. يتأكد أن مدير العائلة يضيف أفرادًا بحسابات منفصلة دون منحهم صلاحية تعديل الاشتراك الرئيسي.
3. يراقب أن كل فرد يستخدم حصته فقط ولا يعدّل الخطة.
4. يدعم عمليات الفصل: تأكيد → اختيار تحويل لحساب مستقل أو ترقية لاشتراك فردي → رسالة نجاح.
5. يتدخل في النزاعات (حصص/فصل) عند تصعيدها له.
6. يربط بيانات العائلة بالتقارير والمحاسبة عند الحاجة.

### الحالات والاستثناءات

- محاولة فرد تعديل الاشتراك الرئيسي → ممنوعة نظاميًا.
- فصل فرد له طلبات نشطة → معالجة الطلبات الجارية قبل/أثناء الفصل.
- ترقية فرد لاشتراك فردي → إنشاء اشتراك مستقل وفق التسعير الحالي (F04).

3. يراقب تغطية كل منطقة (عدد المطاعم/السائقين المتاحين) لتفادي الفجوات.
4. يضبط ما يظهر للعميل: فقط المطاعم/الباقات/العملات المتاحة في منطقته.
5. يتأكد أن المندوب يستقبل فقط الطلبات داخل نطاق دولته الجغرافي مع خرائط محلية.
6. يتخذ قرارات توسعة/تغطية بناءً على الفجوات والطلب.

### الحالات والاستثناءات

- منطقة بلا مطاعم كافية → تظهر محدودية الخيارات؛ يعالجها الأدمن بجذب مطاعم.
- مطعم يلغي تغطية منطقة عليها اشتراكات → معالجة الاشتراكات المتأثرة وتوفير بدائل.
- تداخل حدود مناطق → ضبط دقيق لتفادي ازدواج/ثغرات التغطية.

- بيانات المطعم/الشركة/المالك + الوثائق المرفوعة (السجل التجاري، عقد التأسيس، ترخيص الشركة...).
- تاريخ إصدار وانتهاء كل وثيقة.
- إعدادات التنبيهات الزمنية (شهرين/شهر قبل الانتهاء).

#### لوحة — خطوة 1 _(مستنتجة)_

**محتويات:**

- يستقبل الأدمن طلب تسجيل/تحديث وثائق المطعم مع تواريخها

#### لوحة — خطوة 2 _(مستنتجة)_

**محتويات:**

- يراجع كل وثيقة ويتحقق من صحتها واكتمالها وصلاحية تاريخها

#### لوحة — خطوة 3 _(مستنتجة)_

**محتويات:**

- يعتمد الوثائق (فيظهر المطعم/يستمر) أو يطلب استكمالًا/تصحيحًا

#### لوحة — خطوة 4 _(مستنتجة)_

**محتويات:**

- يتابع التنبيه الآلي قبل انتهاء الوثيقة بشهرين لطلب نسخة جديدة

#### لوحة — خطوة 5 _(مستنتجة)_

**محتويات:**

- يتأكد أنه قبل الانتهاء بشهر يوقف النظام استقبال الطلبات الجديدة ويخفي المطعم ووجباته مؤقتًا

#### لوحة — خطوة 6 _(مستنتجة)_

**محتويات:**

- يعيد التفعيل بعد رفع المطعم نسخة محدّثة واعتمادها

### مسار المستخدم (خطوات)

1. يستقبل الأدمن طلب تسجيل/تحديث وثائق المطعم مع تواريخها.
2. يراجع كل وثيقة ويتحقق من صحتها واكتمالها وصلاحية تاريخها.
3. يعتمد الوثائق (فيظهر المطعم/يستمر) أو يطلب استكمالًا/تصحيحًا.
4. يتابع التنبيه الآلي قبل انتهاء الوثيقة بشهرين لطلب نسخة جديدة.
5. يتأكد أنه قبل الانتهاء بشهر يوقف النظام استقبال الطلبات الجديدة ويخفي المطعم ووجباته مؤقتًا.
6. يعيد التفعيل بعد رفع المطعم نسخة محدّثة واعتمادها.

### الحالات والاستثناءات

- وثيقة منتهية لم تُحدّث → يبقى المطعم مخفيًا حتى الاعتماد.
- وثيقة مزوّرة/غير مطابقة → رفض وتصعيد وإيقاف الحساب.
- تحديث وثيقة أثناء وجود اشتراكات نشطة → ضمان استمرار الطلبات القائمة وفق السياسات.

## F26 — النظام المحاسبي والتسويات

**الهدف:** يدير الأدمن التسويات المالية بين التطبيق والمطاعم على أساس البوكسات الفعلية المُسلّمة، ويتابع الإيرادات والمدفوعات والعمولات والاستردادات والتقارير المالية الشاملة.

**مصدر الشاشات:** مستنتجة من خطوات الـ Workflow

### فلو الشاشات

```mermaid
flowchart TD
  S1["لوحة المتطلبات / المدخلات"]
  S2["لوحة — خطوة 1"]
  S3["صافي مستحق البوكس = سعر البوكس المتفق عليه − (سعر البوكس × نسبة العمولة الديناميكية)"]
  S4["إجمالي المستحقات = (صافي البوكس × عدد البوكسات المُسلّمة) − رسوم اشتراك المطعم (إن استحقت)"]
  S5["لوحة — خطوة 4"]
  S6["لوحة — خطوة 5"]

  S1 --> S2
  S2 --> S3

  S6 --> S7
```

### تفاصيل ومحتويات كل لوحة

#### لوحة المتطلبات / المدخلات _(مستنتجة)_

**محتويات:**

- عدد البوكسات الفعلية المُسلّمة لكل مطعم.
- سعر البوكس المتفق عليه ونسبة العمولة الديناميكية للمطعم (راجع F04).
- رسوم اشتراك المطعم (مبلغ ثابت شهري/سنوي) إن كانت مستحقة.

#### لوحة — خطوة 1 _(مستنتجة)_

**محتويات:**

- يجمّع النظام البوكسات المُسلّمة فعليًا لكل مطعم خلال فترة التسوية

#### صافي مستحق البوكس = سعر البوكس المتفق عليه − (سعر البوكس × نسبة العمولة الديناميكية) _(مستنتجة)_

**محتويات:**

- يحسب الأدمن

#### إجمالي المستحقات = (صافي البوكس × عدد البوكسات المُسلّمة) − رسوم اشتراك المطعم (إن استحقت) _(مستنتجة)_

**محتويات:**

- يحسب

#### لوحة — خطوة 4 _(مستنتجة)_

**محتويات:**

- يخصم أي مبالغ شكاوى محقّة (قيمة بوكس مخصومة من المطعم، راجع F16)

#### لوحة — خطوة 5 _(مستنتجة)_

**محتويات:**

- يعتمد التسوية ويصرف المستحقات للمطعم، ويسجّل العملية

#### يتابع تقارير _(مستنتجة)_

**محتويات:**

- يتابع تقارير: الإيرادات، مدفوعات العملاء، مستحقات المطاعم، العمولات، الاستردادات

### مسار المستخدم (خطوات)

1. يجمّع النظام البوكسات المُسلّمة فعليًا لكل مطعم خلال فترة التسوية.
2. يحسب الأدمن **صافي مستحق البوكس = سعر البوكس المتفق عليه − (سعر البوكس × نسبة العمولة الديناميكية)**.
3. يحسب **إجمالي المستحقات = (صافي البوكس × عدد البوكسات المُسلّمة) − رسوم اشتراك المطعم (إن استحقت)**.
4. يخصم أي مبالغ شكاوى محقّة (قيمة بوكس مخصومة من المطعم، راجع F16).
5. يعتمد التسوية ويصرف المستحقات للمطعم، ويسجّل العملية.
6. يتابع تقارير: الإيرادات، مدفوعات العملاء، مستحقات المطاعم، العمولات، الاستردادات.

### الحالات والاستثناءات

- **مثال:** سعر بوكس 8 د، عمولة 15% → عمولة 1.2، صافي 6.8، 100 بوكس = 680، ناقص اشتراك 50 = **630 د**.
- استرداد عميل (F10) → يُخصم/يُسجّل ضمن الحركة المالية للفترة.
- نزاع تسوية مع مطعم → مراجعة تفصيلية للبوكسات المُسلّمة قبل الصرف.

#### المؤشرات الرئيسية ويختار الفترة الزمنية والنطاق _(مستنتجة)_

**محتويات:**

- يفتح الأدمن لوحة المؤشرات الرئيسية ويختار الفترة الزمنية والنطاق

#### لوحة — خطوة 2 _(مستنتجة)_

**محتويات:**

#### لوحة — خطوة 3 _(مستنتجة)_

**محتويات:**

#### لوحة — خطوة 4 _(مستنتجة)_

**محتويات:**

- يحلّل أداء المطاعم (مبيعات، تقييم، شكاوى) والسائقين (إنجاز، تقييم، زمن)

#### يقارن أداء الدول/المناطق (للـ Super Admin _(مستنتجة)_

**محتويات:**

- يقارن أداء الدول/المناطق (للـ Super Admin: كل الدول أو دولة على حدة)

#### لوحة — خطوة 6 _(مستنتجة)_

**محتويات:**

- يصدّر التقارير ويستخدمها في قرارات التسعير/التغطية/الاستمرارية

### مسار المستخدم (خطوات)

1. يفتح الأدمن لوحة المؤشرات الرئيسية ويختار الفترة الزمنية والنطاق.
2. يتابع مؤشرات الاشتراكات (جديدة/مجدّدة/ملغاة/مجمّدة) والطلبات (مؤكدة/مُسلّمة).
3. يراقب الإيرادات والعمولات والاستردادات (يتقاطع مع F26).
4. يحلّل أداء المطاعم (مبيعات، تقييم، شكاوى) والسائقين (إنجاز، تقييم، زمن).
5. يقارن أداء الدول/المناطق (للـ Super Admin: كل الدول أو دولة على حدة).
6. يصدّر التقارير ويستخدمها في قرارات التسعير/التغطية/الاستمرارية.

### الحالات والاستثناءات

- هبوط مؤشر حرج (إيراد/تقييم/إنجاز) → تنبيه واستقصاء السبب.
- بيانات ناقصة لفترة → توضيح الفجوة وعدم بناء قرار عليها.
- اختلاف نطاق الصلاحية → Country Admin لا يرى بيانات دول أخرى.

## F28 — التوسع الدولي و Multi-Tenancy (Super Admin / Country Admin)

**الهدف:** ينظّم هذا التدفق التوسع الدولي عبر طبقتين من الأدمن: **Super Admin** يدير الدول والإعدادات العالمية والرقابة الشاملة، و**Country Admin** يدير دولته فقط، مع عزل تام لبيانات كل دولة.

**مصدر الشاشات:** مستنتجة من خطوات الـ Workflow

### فلو الشاشات

```mermaid
flowchart TD
  S1["لوحة المتطلبات / المدخلات"]
  S2["Super Admin:"]
  S3["Super Admin:"]
  S4["Super Admin:"]
  S5["Country Admin:"]
  S6["يتأكد الطرفان من عزل البيانات"]
  S7["Super Admin:"]
  S1 --> S2
  S2 --> S3
  S3 --> S4
  S4 --> S5
  S5 --> S6
  S6 --> S7
```

### تفاصيل ومحتويات كل لوحة

#### لوحة المتطلبات / المدخلات _(مستنتجة)_

**محتويات:**

- صلاحيات مفصولة: Super Admin (عالمي) مقابل Country Admin (نطاق دولة واحدة).
- إعدادات كل دولة: العملة، بوابات الدفع المحلية، اللغات.
- بنية عزل بيانات تضمن خصوصية وأداء كل دولة.

#### Super Admin: _(مستنتجة)_

**محتويات:**

- يضيف/يفعّل دولة جديدة ويضبط إعداداتها الأساسية (عملة، لغات، بوابة دفع)

#### Super Admin: _(مستنتجة)_

**محتويات:**

- يعيّن Country Admin لكل دولة ويمنحه صلاحيات نطاقه فقط

#### Super Admin: _(مستنتجة)_

**محتويات:**

- يراقب تقارير الأداء/الإيرادات/المشتركين لكل الدول أو دولة على حدة، ويضبط الإعدادات العالمية

#### Country Admin: _(مستنتجة)_

**محتويات:**

- يدير داخل دولته فقط: المطاعم، السائقين، العمولات، الاشتراكات، الشكاوى (كل الميزات F01–F31 ضمن نطاقه)

#### يتأكد الطرفان من عزل البيانات _(مستنتجة)_

**محتويات:**

- يتأكد الطرفان من عزل البيانات: لا تداخل بين بيانات الدول ولا بين العمليات البنكية

#### Super Admin: _(مستنتجة)_

**محتويات:**

- يخصّص اللغات/اللهجات لكل دولة (يتقاطع مع F29)

### مسار المستخدم (خطوات)

1. **Super Admin:** يضيف/يفعّل دولة جديدة ويضبط إعداداتها الأساسية (عملة، لغات، بوابة دفع).
2. **Super Admin:** يعيّن Country Admin لكل دولة ويمنحه صلاحيات نطاقه فقط.
3. **Super Admin:** يراقب تقارير الأداء/الإيرادات/المشتركين لكل الدول أو دولة على حدة، ويضبط الإعدادات العالمية.
4. **Country Admin:** يدير داخل دولته فقط: المطاعم، السائقين، العمولات، الاشتراكات، الشكاوى (كل الميزات F01–F31 ضمن نطاقه).
5. يتأكد الطرفان من عزل البيانات: لا تداخل بين بيانات الدول ولا بين العمليات البنكية.
6. **Super Admin:** يخصّص اللغات/اللهجات لكل دولة (يتقاطع مع F29).

### الحالات والاستثناءات

- محاولة Country Admin الوصول لبيانات دولة أخرى → ممنوعة نظاميًا (عزل صارم).
- إطلاق دولة بلا بوابة دفع/عملة مكتملة → لا تُفعّل للعملاء حتى اكتمال الإعداد.
- تعارض إعداد عالمي مع خصوصية دولة → الإعداد المحلي يُحترم ضمن حدود السياسة العامة.
