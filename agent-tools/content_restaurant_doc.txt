# تفاصيل الميزات

## F01 — تسجيل المطعم واعتماد الحساب والدخول

**الهدف:** تسجيل المطعم رسميًا على المنصة عبر لوحة تحكم ويب (وليس تطبيق موبايل)، برفع بيانات الشركة الرسمية ووثائقها، ثم انتظار مراجعة الأدمن واعتماده.
لا يُفعّل الحساب ولا يظهر المطعم للعملاء إلا بعد موافقة الأدمن.

**مصدر الشاشات:** مستنتجة من خطوات الـ Workflow

### فلو الشاشات

```mermaid
flowchart TD
  S1["لوحة المتطلبات / المدخلات"]
  S2["تسجيل مطعم جديد"]
  S3["لوحة — خطوة 2"]
  S4["لوحة — خطوة 3"]
  S5["لوحة — خطوة 4"]
  S6["قيد المراجعة"]
  S7["لوحة — خطوة 6"]
  S8["عند الموافقة"]
  S9["نسيت كلمة المرور"]
  S1 --> S2
  S2 --> S3
  S3 --> S4
  S4 --> S5
  S5 --> S6
  S6 --> S7
  S7 --> S8
  S8 --> S9
```

### تفاصيل ومحتويات كل لوحة

#### لوحة المتطلبات / المدخلات _(مستنتجة)_

**محتويات:**

- بيانات الشركة الرسمية وبيانات المالك (مرجع F22).
- وثائق إلزامية: السجل التجاري، عقد التأسيس، ترخيص الشركة، وأي وثائق مطلوبة بتواريخ إصدار وانتهاء.
- بريد إلكتروني/هاتف وكلمة مرور خاصة بلوحة المطعم.
- المحافظات/المناطق المخدومة مبدئيًا (مرجع F21).

#### تسجيل مطعم جديد _(مستنتجة)_

**محتويات:**

- فتح صفحة  على لوحة الويب

#### لوحة — خطوة 2 _(مستنتجة)_

**محتويات:**

- إدخال بيانات الشركة والمالك والنشاط

#### لوحة — خطوة 3 _(مستنتجة)_

**محتويات:**

- رفع الوثائق المطلوبة مع تواريخ الإصدار والانتهاء

#### لوحة — خطوة 4 _(مستنتجة)_

**محتويات:**

- تحديد المناطق المخدومة مبدئيًا

#### قيد المراجعة _(مستنتجة)_

**محتويات:**

- إرسال الطلب → ينتقل الحساب إلى حالة

#### لوحة — خطوة 6 _(مستنتجة)_

**محتويات:**

- مراجعة الأدمن للبيانات والوثائق

#### عند الموافقة _(مستنتجة)_

**محتويات:**

- عند الموافقة: تفعيل الحساب وإشعار المطعم، ويصبح ظاهرًا للعملاء بعد إكمال الأسعار والقوائم المعتمدة

#### نسيت كلمة المرور _(مستنتجة)_

**محتويات:**

- الدخول لاحقًا عبر تسجيل دخول خاص (Email/Phone + Password) مع خيار

### مسار المستخدم (خطوات)

1. فتح صفحة "تسجيل مطعم جديد" على لوحة الويب.
2. إدخال بيانات الشركة والمالك والنشاط.
3. رفع الوثائق المطلوبة مع تواريخ الإصدار والانتهاء.
4. تحديد المناطق المخدومة مبدئيًا.
5. إرسال الطلب → ينتقل الحساب إلى حالة "قيد المراجعة".
6. مراجعة الأدمن للبيانات والوثائق.
7. عند الموافقة: تفعيل الحساب وإشعار المطعم، ويصبح ظاهرًا للعملاء بعد إكمال الأسعار والقوائم المعتمدة.
8. الدخول لاحقًا عبر تسجيل دخول خاص (Email/Phone + Password) مع خيار "نسيت كلمة المرور".

### الحالات والاستثناءات

- رفض الأدمن بسبب نقص/خطأ وثيقة → إشعار بالسبب وإعادة الرفع.
- الحساب "قيد المراجعة" لا يستقبل طلبات ولا يظهر للعملاء.
- وثيقة منتهية أو ناقصة توقف الاعتماد (مرجع F22).
- نسيان كلمة المرور → استرجاع آمن عبر اللوحة.

## F02 — تسعير البرامج والباقات لمدة 26 يوم

**الهدف:** يحدد المطعم سعره الخاص لكل برنامج غذائي وكل باقة لمدة 26 يوم عمل من لوحته.
هذه الأسعار هي الأساس الذي يبني عليه النظام تصنيف المطعم وقيمة البوكس اليومي ومتوسطات القروب.

**مصدر الشاشات:** مستنتجة من خطوات الـ Workflow

### فلو الشاشات

```mermaid
flowchart TD
  S1["لوحة المتطلبات / المدخلات"]
  S2["التسعير"]
  S3["لوحة — خطوة 2"]
  S4["لوحة — خطوة 3"]
  S5["لوحة — خطوة 4"]
  S6["قيمة البوكس اليومي"]
  S7["لوحة — خطوة 6"]
  S8["لوحة — خطوة 7"]
  S1 --> S2
  S2 --> S3
  S3 --> S4
  S4 --> S5
  S5 --> S6
  S6 --> S7
  S7 --> S8
```

### تفاصيل ومحتويات كل لوحة

#### لوحة المتطلبات / المدخلات _(مستنتجة)_

**محتويات:**

- حساب مطعم معتمد (F01).
- قائمة البرامج الغذائية (نزول وزن، ضخامة عضلية، محافظة، كيتو) والباقات (كاملة، غداء، مخصصة) كما يحددها الأدمن.
- بيانات تكلفة المطعم لتسعير عادل.

#### التسعير _(مستنتجة)_

**محتويات:**

- فتح شاشة  في لوحة المطعم

#### لوحة — خطوة 2 _(مستنتجة)_

**محتويات:**

- اختيار البرنامج الغذائي المطلوب تسعيره

#### لوحة — خطوة 3 _(مستنتجة)_

**محتويات:**

- اختيار الباقة المرتبطة

#### لوحة — خطوة 4 _(مستنتجة)_

**محتويات:**

- إدخال سعر اشتراك 26 يوم لكل تركيبة (برنامج × باقة)

#### قيمة البوكس اليومي _(مستنتجة)_

**محتويات:**

- مراجعة  المحسوبة تلقائيًا = السعر ÷ 26

#### لوحة — خطوة 6 _(مستنتجة)_

**محتويات:**

- حفظ الأسعار وإرسالها

#### لوحة — خطوة 7 _(مستنتجة)_

**محتويات:**

- النظام يستخدم الأسعار في التصنيف (F03) ومتوسط القروب (F04)

### مسار المستخدم (خطوات)

1. فتح شاشة "التسعير" في لوحة المطعم.
2. اختيار البرنامج الغذائي المطلوب تسعيره.
3. اختيار الباقة المرتبطة.
4. إدخال سعر اشتراك 26 يوم لكل تركيبة (برنامج × باقة).
5. مراجعة "قيمة البوكس اليومي" المحسوبة تلقائيًا = السعر ÷ 26.
6. حفظ الأسعار وإرسالها.
7. النظام يستخدم الأسعار في التصنيف (F03) ومتوسط القروب (F04).

### الحالات والاستثناءات

- برنامج/باقة بدون سعر → لا يظهر المطعم ضمن تلك التركيبة للعملاء.
- تعديل أي سعر يعيد حساب التصنيف والمتوسطات تلقائيًا.
- قد يضيف الأدمن أو يلغي برامج/باقات ديناميكيًا → تتحدّث شاشة التسعير لدى المطعم.

## F03 — التصنيف التلقائي للمطعم

**الهدف:** يصنّف النظام المطعم تلقائيًا (Basic / Platinum / Elite) بناءً على سعر اشتراكه لـ26 يوم مقارنة بالحدود السعرية التي يضعها الأدمن.
التصنيف يحدد القروب الذي يظهر فيه المطعم للعملاء، وهو تصنيف داخلي بين الأدمن والمطاعم.

**مصدر الشاشات:** مستنتجة من خطوات الـ Workflow

### فلو الشاشات

```mermaid
flowchart TD
  S1["لوحة المتطلبات / المدخلات"]
  S2["لوحة — خطوة 1"]
  S3["يقارن السعر بالحدود (مثال"]
  S4["لوحة — خطوة 3"]
  S5["المطعم"]
  S6["لوحة — خطوة 5"]
  S1 --> S2
  S2 --> S3
  S3 --> S4
  S4 --> S5
  S5 --> S6
```

### تفاصيل ومحتويات كل لوحة

#### لوحة المتطلبات / المدخلات _(مستنتجة)_

**محتويات:**

- إدخال أسعار اشتراك 26 يوم (F02).
- الحدود السعرية للتصنيفات المعرّفة من الأدمن.

#### لوحة — خطوة 1 _(مستنتجة)_

**محتويات:**

- بعد حفظ الأسعار، يحسب النظام سعر اشتراك 26 يوم للمطعم

#### يقارن السعر بالحدود (مثال _(مستنتجة)_

**محتويات:**

- يقارن السعر بالحدود (مثال: Basic < 100 د.ك، Platinum < 150، Elite < 200)

#### لوحة — خطوة 3 _(مستنتجة)_

**محتويات:**

- يحدد النظام تصنيف المطعم تلقائيًا

#### المطعم _(مستنتجة)_

**محتويات:**

- يعرض التصنيف الحالي في لوحة المطعم

#### لوحة — خطوة 5 _(مستنتجة)_

**محتويات:**

- عند تغيير المطعم لأسعاره أو تغيير الأدمن للحدود → إعادة تصنيف تلقائية

### مسار المستخدم (خطوات)

1. بعد حفظ الأسعار، يحسب النظام سعر اشتراك 26 يوم للمطعم.
2. يقارن السعر بالحدود (مثال: Basic < 100 د.ك، Platinum < 150، Elite < 200).
3. يحدد النظام تصنيف المطعم تلقائيًا.
4. يعرض التصنيف الحالي في لوحة المطعم.
5. عند تغيير المطعم لأسعاره أو تغيير الأدمن للحدود → إعادة تصنيف تلقائية.

### الحالات والاستثناءات

- تعديل حدود الأدمن قد ينقل المطعم إلى تصنيف أعلى أو أدنى.
- التدرّج تراكمي: Platinum يضم Basic، و Elite يضم الجميع → قد يظهر المطعم لعملاء تصنيفات أعلى.
- لا يستطيع المطعم تغيير تصنيفه يدويًا؛ يتم فقط عبر السعر.

## F04 — التسعير والعمولة الديناميكية على المطعم

**الهدف:** توضيح كيف تُحتسب العمولة الديناميكية الخاصة بالمطعم وتُخصم من «سعر البوكس المتفق عليه» بين المنصة والمطعم — وليس من سعر العميل.
يرى المطعم نسبته الخاصة وصافي مستحقه المتوقع لكل بوكس.

**مصدر الشاشات:** مستنتجة من خطوات الـ Workflow

### فلو الشاشات

```mermaid
flowchart TD
  S1["لوحة المتطلبات / المدخلات"]
  S2["يحدد الأدمن نسبة عمولة خاصة بالمطعم (مثال"]
  S3["لوحة — خطوة 2"]
  S4["لكل بوكس مُسلّم"]
  S5["لوحة — خطوة 4"]
  S6["لوحة — خطوة 5"]
  S1 --> S2
  S2 --> S3
  S3 --> S4
  S4 --> S5
  S5 --> S6
```

### تفاصيل ومحتويات كل لوحة

#### لوحة المتطلبات / المدخلات _(مستنتجة)_

**محتويات:**

- نسبة عمولة المطعم التي يحددها الأدمن (تختلف من مطعم لآخر).
- سعر البوكس المتفق عليه بين المنصة والمطعم.

#### يحدد الأدمن نسبة عمولة خاصة بالمطعم (مثال _(مستنتجة)_

**محتويات:**

- يحدد الأدمن نسبة عمولة خاصة بالمطعم (مثال: 15%)

#### لوحة — خطوة 2 _(مستنتجة)_

**محتويات:**

- يطّلع المطعم على نسبته الحالية في لوحته

#### لكل بوكس مُسلّم _(مستنتجة)_

**محتويات:**

- لكل بوكس مُسلّم: صافي مستحق المطعم = سعر البوكس − (سعر البوكس × النسبة)

#### لوحة — خطوة 4 _(مستنتجة)_

**محتويات:**

- مراجعة الصافي المتوقع لكل بوكس قبل التسوية

#### لوحة — خطوة 5 _(مستنتجة)_

**محتويات:**

- متابعة الإجمالي ضمن التسويات (F26)

### مسار المستخدم (خطوات)

1. يحدد الأدمن نسبة عمولة خاصة بالمطعم (مثال: 15%).
2. يطّلع المطعم على نسبته الحالية في لوحته.
3. لكل بوكس مُسلّم: صافي مستحق المطعم = سعر البوكس − (سعر البوكس × النسبة).
4. مراجعة الصافي المتوقع لكل بوكس قبل التسوية.
5. متابعة الإجمالي ضمن التسويات (F26).

### الحالات والاستثناءات

- تغيير الأدمن للنسبة يسري على البوكسات اللاحقة فقط.
- العمولة لا تمس سعر العميل النهائي إطلاقًا.
- قد تختلف النسبة بين مطعم وآخر (مثال: مطعم A 15%، مطعم B 20%).

## F05 — استقبال اختيارات التقويم كطلبات يومية

**الهدف:** يستقبل المطعم ناتج اختيارات العملاء من التقويم الذكي كطلبات يومية مؤكدة عند بدء نافذة 48 ساعة.
المطعم لا يستخدم التقويم نفسه (فهو أداة العميل)، بل يرى الطلبات الناتجة عنه في لوحته.

**مصدر الشاشات:** مستنتجة من خطوات الـ Workflow

### فلو الشاشات

```mermaid
flowchart TD
  S1["لوحة المتطلبات / المدخلات"]
  S2["لوحة — خطوة 1"]
  S3["لوحة — خطوة 2"]
  S4["لوحة — خطوة 3"]
  S5["يعرض الطلب"]
  S6["لوحة — خطوة 5"]
  S1 --> S2
  S2 --> S3
  S3 --> S4
  S4 --> S5
  S5 --> S6
```

### تفاصيل ومحتويات كل لوحة

#### لوحة المتطلبات / المدخلات _(مستنتجة)_

**محتويات:**

- حساب معتمد وقوائم معتمدة (F24).
- تحديد الطاقة الاستيعابية اليومية (F08).
- دخول الطلب في نافذة 48 ساعة قبل التوصيل.

#### لوحة — خطوة 1 _(مستنتجة)_

**محتويات:**

- يختار العميل اليوم والمطعم والبوكس من تطبيقه

#### لوحة — خطوة 2 _(مستنتجة)_

**محتويات:**

- عند بلوغ 48 ساعة قبل التوصيل يتثبّت الاختيار ويُرسَل كطلب للمطعم

#### لوحة — خطوة 3 _(مستنتجة)_

**محتويات:**

- يرى المطعم الطلبات اليومية الجديدة في لوحته

#### يعرض الطلب _(مستنتجة)_

**محتويات:**

- يعرض الطلب: عدد البوكسات، نوعها، وقت التوصيل المطلوب، والموقع العام فقط

#### لوحة — خطوة 5 _(مستنتجة)_

**محتويات:**

- يبدأ المطعم التحضير ويُصدر الفواتير والملصقات (F12)

### مسار المستخدم (خطوات)

1. يختار العميل اليوم والمطعم والبوكس من تطبيقه.
2. عند بلوغ 48 ساعة قبل التوصيل يتثبّت الاختيار ويُرسَل كطلب للمطعم.
3. يرى المطعم الطلبات اليومية الجديدة في لوحته.
4. يعرض الطلب: عدد البوكسات، نوعها، وقت التوصيل المطلوب، والموقع العام فقط.
5. يبدأ المطعم التحضير ويُصدر الفواتير والملصقات (F12).

### الحالات والاستثناءات

- قبل 48 ساعة لا تظهر اختيارات مؤكدة لأنها قابلة للتغيير من العميل.
- قد يصل طلب ناتج عن اختيار تلقائي للنظام (F07) ويُعامل كطلب عادي.
- قد يعدّل الأدمن طلبًا داخل 48 ساعة كاستثناء طارئ عبر واتساب.

## F06 — نافذة التحضير 48 ساعة

**الهدف:** تثبيت طلبات العملاء قبل 48 ساعة من التوصيل لتبدأ مرحلة التحضير لدى المطعم دون أي تعديل من العميل.
عند بدء النافذة تُرسل الطلبات وتتولّد الفواتير والملصقات تلقائيًا.

**مصدر الشاشات:** مستنتجة من خطوات الـ Workflow

### فلو الشاشات

```mermaid
flowchart TD
  S1["لوحة المتطلبات / المدخلات"]
  S2["لوحة — خطوة 1"]
```

#### لوحة — خطوة 1 _(مستنتجة)_

**محتويات:**

- يعدّ النظام عدد البوكسات المُسلّمة فعليًا

#### لوحة — خطوة 2 _(مستنتجة)_

**محتويات:**

- حساب صافي كل بوكس = سعر البوكس − (سعر البوكس × نسبة العمولة)

#### لوحة — خطوة 3 _(مستنتجة)_

**محتويات:**

- إجمالي المستحقات = (صافي البوكس × عدد البوكسات المُسلّمة) − رسوم اشتراك المطعم (إن استُحقت)

#### لوحة — خطوة 4 _(مستنتجة)_

**محتويات:**

- خصم قيمة بوكسات الشكاوى المحقّة (F16)

#### لوحة — خطوة 5 _(مستنتجة)_

**محتويات:**

- عرض كشف التسوية في اللوحة

#### لوحة — خطوة 6 _(مستنتجة)_

**محتويات:**

- استلام المستحقات حسب الدورة المتفق عليها

### مسار المستخدم (خطوات)

1. يعدّ النظام عدد البوكسات المُسلّمة فعليًا.
2. حساب صافي كل بوكس = سعر البوكس − (سعر البوكس × نسبة العمولة).
3. إجمالي المستحقات = (صافي البوكس × عدد البوكسات المُسلّمة) − رسوم اشتراك المطعم (إن استُحقت).
4. خصم قيمة بوكسات الشكاوى المحقّة (F16).
5. عرض كشف التسوية في اللوحة.
6. استلام المستحقات حسب الدورة المتفق عليها.

### الحالات والاستثناءات

- رسوم اشتراك المطعم (مبلغ ثابت شهري/سنوي) تُحصّل مباشرة أو تُخصم من المستحقات.
- خصومات الشكاوى المحقّة تظهر بندًا في الكشف.
- تسوية نهائية عند إنهاء التعاقد (F25).

#### لوحة — خطوة 5 _(مستنتجة)_

**محتويات:**

#### لوحة — خطوة 6 _(مستنتجة)_

- مراجعة/تصدير التقارير دوريًا

### الحالات والاستثناءات

- بيانات الشكاوى تنعكس على مؤشرات الجودة (KPI).
- التقارير محصورة ببيانات المطعم نفسه فقط.

## F31 — حدود بيانات العميل الظاهرة للمطعم

**الهدف:** ضمان خصوصية العميل بعدم كشف بياناته الشخصية للمطعم.
يرى المطعم فقط ما يلزم لتجهيز البوكسات وتسليمها، دون اسم العميل أو رقمه أو عنوانه الدقيق.

**مصدر الشاشات:** مستنتجة من خطوات الـ Workflow

### فلو الشاشات

```mermaid
flowchart TD
  S1["لوحة المتطلبات / المدخلات"]
  S2["لوحة — خطوة 1"]
  S3["الاطلاع على"]
  S4["لوحة — خطوة 3"]
  S5["لوحة — خطوة 4"]
  S6["لوحة — خطوة 5"]
  S1 --> S2
  S2 --> S3
  S3 --> S4
  S4 --> S5
  S5 --> S6
```

### تفاصيل ومحتويات كل لوحة

#### لوحة المتطلبات / المدخلات _(مستنتجة)_

**محتويات:**

- نظام طلبات يعرض بيانات محدودة فقط.
- التزام المطعم بمعايير الأمان والخصوصية.

#### لوحة — خطوة 1 _(مستنتجة)_

**محتويات:**

- استقبال الطلب ببيانات محدودة فقط

#### الاطلاع على _(مستنتجة)_

**محتويات:**

- الاطلاع على: عدد البوكسات، نوعها، الموقع العام للتوصيل، وقت التوصيل المطلوب

#### لوحة — خطوة 3 _(مستنتجة)_

**محتويات:**

- تجهيز البوكسات دون معرفة هوية العميل

#### لوحة — خطوة 4 _(مستنتجة)_

**محتويات:**

- تسليم البوكسات للسائق المعتمد (F13)

#### لوحة — خطوة 5 _(مستنتجة)_

**محتويات:**

- أي تواصل لازم يتم عبر قنوات المنصة الآمنة فقط

### مسار المستخدم (خطوات)

1. استقبال الطلب ببيانات محدودة فقط.
2. الاطلاع على: عدد البوكسات، نوعها، الموقع العام للتوصيل، وقت التوصيل المطلوب.
3. تجهيز البوكسات دون معرفة هوية العميل.
4. تسليم البوكسات للسائق المعتمد (F13).
5. أي تواصل لازم يتم عبر قنوات المنصة الآمنة فقط.

### الحالات والاستثناءات

- لا يظهر اسم العميل أو رقمه أو عنوانه الدقيق للمطعم إطلاقًا.
- أي إظهار لبيانات إضافية = حالة استثنائية موثّقة بإشعار للأدمن.
- الشكاوى تُعرض للمطعم دون كشف هوية مفصّلة للعميل.
