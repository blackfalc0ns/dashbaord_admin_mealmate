# F001 — نقاط الضعف التفصيلية

## الميزة
**Customer Registration, Login & Onboarding**  
الميزة مسؤولة عن أول احتكاك بين العميل وMealMate: التعريف بالخدمة، إنشاء الحساب، تسجيل الدخول، والتحقق الأولي قبل الدخول لمسار الاشتراك.

## 1. نقاط ضعف من ناحية الـ Business
### 1.1 عدم فصل الـ Onboarding عن التسجيل
الملف الحالي يتعامل مع onboarding والتسجيل والدخول كأنهم Feature واحدة. Business-wise هذا يسبب خلط بين هدفين مختلفين:
- الـ Onboarding هدفه إقناع العميل وفهم القيمة.
- التسجيل هدفه إنشاء حساب آمن.
- الدخول هدفه إدارة جلسة وصلاحية.

الأثر: صعوبة قياس conversion funnel بدقة: هل العميل ترك التطبيق بسبب الرسائل التسويقية، طول نموذج التسجيل، أم فشل OTP؟

### 1.2 عدم تحديد استراتيجية التحقق من العميل
لا يوجد قرار واضح: هل التحقق عبر الهاتف؟ البريد؟ الاثنين؟ هل OTP إلزامي قبل الاشتراك أم قبل الدفع؟

الأثر: قد تظهر حسابات وهمية أو duplicate accounts، وقد يتأثر الدفع والمحفظة والاشتراك لاحقًا.

### 1.3 غياب تعريف شرائح العملاء
لا يوجد تفصيل لاختلاف التسجيل بين:
- عميل فردي.
- عميل عائلي.
- عميل قادم من referral أو influencer code.
- عميل من دولة أو منطقة غير مدعومة.

الأثر: تجربة واحدة عامة قد لا تخدم نموذج الاشتراك العائلي أو حملات النمو.

### 1.4 لا يوجد تعريف واضح للنجاح التجاري
مؤشرات الأداء الحالية عامة. المطلوب Business KPIs محددة مثل:
- registration completion rate.
- OTP success rate.
- onboarding skip rate.
- first subscription conversion.
- login failure rate.

الأثر: صعوبة تحسين رحلة العميل بعد الإطلاق.

## 2. نقاط ضعف من ناحية الـ Logic / منطق التشغيل
### 2.1 State Machine غير محددة
الحالات الحالية عامة مثل `Draft`, `Pending`, `Active`. العميل يحتاج حالات أدق:
- `Guest`
- `Registered`
- `PendingVerification`
- `Verified`
- `Active`
- `Suspended`
- `Closed`

الأثر: قد يدخل العميل لمسارات لا يجب الوصول لها، مثل الدفع قبل التحقق أو استخدام المحفظة قبل تفعيل الحساب.

### 2.2 منطق منع التكرار غير مفصل
التوثيق يذكر منع التكرار، لكنه لا يحدد هل التكرار حسب:
- Email
- Phone
- Device
- National ID لاحقًا
- Social provider ID

الأثر: عميل واحد قد يمتلك أكثر من حساب ويستغل العروض أو referral rewards.

### 2.3 منطق الجلسات غير موجود
لا يوجد تعريف لـ:
- refresh token.
- مدة الجلسة.
- تسجيل الخروج من كل الأجهزة.
- التعامل مع تغيير كلمة المرور.
- ماذا يحدث عند تعطيل الحساب.

الأثر: مخاطر أمنية وتجربة غير مستقرة عند تغيير بيانات الحساب.

### 2.4 منطق نسيان كلمة المرور ناقص
لا توجد خطوات واضحة لإعادة تعيين كلمة المرور، صلاحية OTP، عدد المحاولات، وإلغاء الرموز القديمة.

الأثر: قابلية استغلال OTP أو إرباك المستخدم عند فشل الاستعادة.

## 3. نقاط ضعف Data / API / Security
### 3.1 API endpoints عامة وغير طبيعية
المسار الحالي مثل `customer_registration_login_onboarding` طويل وغير مناسب كتصميم API. الأفضل فصل:
- `POST /auth/register/customer`
- `POST /auth/login`
- `POST /auth/verify-otp`
- `POST /auth/refresh`
- `POST /auth/forgot-password`

### 3.2 لا يوجد schema للبيانات
لا يوجد تحديد للحقول المطلوبة: الاسم، الهاتف، البريد، الدولة، المنطقة، اللغة، كلمة المرور، نوع الحساب.

### 3.3 سياسات الأمان غير مفصلة
ينقص:
- password policy.
- rate limiting.
- OTP expiry.
- brute force protection.
- device/session tracking.
- account lockout policy.

## 4. أثر نقاط الضعف
- زيادة fake/duplicate accounts.
- صعوبة قياس conversion.
- مخاطر على العروض والإحالات.
- ثغرات في OTP أو الجلسات.
- اختلاف تنفيذ الـ Frontend والـ Backend.

## 5. المطلوب لتقوية الميزة
- فصل onboarding عن auth flows.
- تعريف customer account state machine.
- اعتماد OTP policy واضحة.
- تصميم APIs قصيرة ومباشرة.
- إضافة funnel KPIs.
- توثيق schemas وvalidation messages.
