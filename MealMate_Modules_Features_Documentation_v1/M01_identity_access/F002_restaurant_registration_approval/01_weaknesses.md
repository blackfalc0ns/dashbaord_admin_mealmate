# F002 — نقاط الضعف التفصيلية

## الميزة
**Restaurant Registration & Approval**  
الميزة مسؤولة عن استقبال طلبات انضمام المطاعم، مراجعة بياناتها ووثائقها، ثم اعتمادها للتشغيل داخل MealMate.

## 1. نقاط ضعف من ناحية الـ Business
### 1.1 عدم وضوح معايير قبول المطعم
التوثيق لا يحدد ما الذي يجعل المطعم مقبولًا أو مرفوضًا:
- هل يجب وجود رخصة تجارية؟
- هل يجب حد أدنى للتقييم أو جودة الطعام؟
- هل المطعم يجب أن يغطي مناطق معينة؟
- هل القبول مرتبط بسعة يومية أو نوع برنامج غذائي؟

الأثر: قرارات الاعتماد قد تصبح شخصية وغير موحدة بين الدول أو الأدمنز.

### 1.2 عدم ربط التسجيل بنموذج التشغيل
المطعم في MealMate ليس مجرد vendor. هو جزء من منظومة اشتراكات، وجبات يومية، طاقة استيعابية، مناطق، وفواتير.

نقطة الضعف: التسجيل لا يطلب صراحة بيانات تشغيلية مهمة مثل:
- البرامج المدعومة.
- الباقات والأسعار.
- الطاقة اليومية.
- مناطق الخدمة.
- مواعيد التحضير.
- هل لديه مناديب أم يعتمد على مناديب MealMate؟

الأثر: مطعم قد يتم اعتماده Business-wise لكنه غير جاهز عمليًا لاستقبال الطلبات.

### 1.3 لا يوجد SLA للمراجعة
لا يوجد زمن واضح لمراجعة طلب مطعم جديد أو طلب استكمال وثائق.

الأثر: تأخير onboarding للمطاعم، وفقدان شركاء محتملين، وعدم وضوح المسؤولية التشغيلية.

### 1.4 عدم وجود مراحل نضج للمطعم
التوثيق يقف عند الاعتماد، لكنه لا يحدد ما بعد الاعتماد:
- `Approved but not configured`
- `Configured`
- `Ready for orders`
- `Live`

الأثر: احتمال تفعيل مطعم قبل اكتمال التسعير أو القوائم أو الطاقة.

## 2. نقاط ضعف من ناحية الـ Logic / منطق التشغيل
### 2.1 State Machine عامة
الحالات الحالية لا تكفي. المطعم يحتاج حالات مثل:
- `Draft`
- `PendingDocuments`
- `PendingAdminReview`
- `Rejected`
- `Approved`
- `NeedsOperationalSetup`
- `ReadyToGoLive`
- `Active`
- `Suspended`
- `Terminated`

الأثر: الانتقال من التسجيل إلى التشغيل قد يحدث بدون خطوات إلزامية.

### 2.2 منطق الرفض وإعادة التقديم غير محدد
لا يوجد تفصيل لـ:
- أسباب الرفض.
- هل يمكن إعادة التقديم؟
- هل تعاد مراجعة كل الوثائق أم الوثيقة المرفوضة فقط؟
- ما عدد مرات إعادة التقديم؟

الأثر: تجربة سيئة للمطعم وصعوبة متابعة الطلبات.

### 2.3 علاقة F002 مع F007 غير محكمة
اعتماد المطعم يجب أن يعتمد على Document Verification. حاليًا العلاقة مذكورة ضمنيًا فقط.

الأثر: قد يتم اعتماد مطعم دون اكتمال وثائقه أو دون وضوح حالة كل وثيقة.

### 2.4 لا يوجد منطق واضح لـ Restaurant Owner
تسجيل المطعم يحتاج ربط حساب مالك أو مدير مسؤول. التوثيق لا يوضح:
- هل أول مستخدم يصبح owner؟
- هل يمكن وجود أكثر من owner؟
- من يستطيع إضافة مستخدمين للمطعم؟

الأثر: مشاكل ملكية وصلاحيات داخل لوحة المطعم.

## 3. نقاط ضعف Data / API / Security
### 3.1 نموذج البيانات ناقص
ينقص كيانات واضحة مثل:
- `RestaurantRegistration`
- `RestaurantProfile`
- `RestaurantOwner`
- `RestaurantDocument`
- `RestaurantApprovalDecision`
- `ServiceRegion`
- `OperationalReadinessChecklist`

### 3.2 API لا يعكس مسار الاعتماد
المسارات العامة لا تكفي. المطلوب:
- submit registration.
- upload documents.
- request changes.
- approve.
- reject.
- mark operationally ready.

### 3.3 غياب audit تفصيلي لقرار الاعتماد
قرار قبول/رفض مطعم يجب أن يسجل:
- من اتخذ القرار.
- وقت القرار.
- السبب.
- الوثائق التي تم الاعتماد عليها.
- before/after للحالة.

## 4. أثر نقاط الضعف
- اعتماد مطاعم غير جاهزة.
- تفاوت قرارات الأدمن.
- تأخير تشغيل المطاعم.
- نزاعات عند رفض أو تعليق مطعم.
- فتح مخاطر قانونية وتشغيلية.

## 5. المطلوب لتقوية الميزة
- تعريف approval criteria موحدة.
- ربط F002 إلزاميًا بـ F007.
- إضافة Operational Readiness Checklist.
- تعريف state machine خاصة بالمطاعم.
- إضافة SLA للمراجعة.
- فصل approval عن go-live.
