# F002 — Feature Analysis

## Context
ميزة **Restaurant Registration & Approval** مسؤولة عن تسجيل المطعم كشريك داخل MealMate، وجمع بياناته الرسمية والتشغيلية، ثم إرسال الطلب للأدمن للمراجعة والقبول أو الرفض.

على عكس العميل، المطعم **لا يدخل التشغيل مباشرة**. المطعم يقدر يبدأ طلب التسجيل ويدخل بياناته ويرفع مستنداته، لكن لا يصبح `Active` ولا يظهر للعملاء إلا بعد مراجعة الأدمن واعتماد الطلب.

## Business Analysis
### تسجيل المطعم يحتاج مراجعة
المطعم كيان تجاري وشريك تشغيل، لذلك لازم يتم التحقق منه قبل السماح له باستقبال طلبات أو الظهور في تقويم العملاء.

الأسباب:
- حماية MealMate من مطاعم غير مرخصة.
- التأكد من هوية المالك أو المسؤول.
- التأكد من السجل التجاري والترخيص.
- تقليل مخاطر الشكاوى أو النزاعات القانونية.
- ضمان جاهزية المطعم قبل التشغيل.

### البيانات المطلوبة من المطعم
يجب أن يدخل المطعم بيانات أساسية قبل الإرسال للمراجعة:
- اسم المطعم التجاري.
- اسم الشركة المالكة إن وجد.
- الدولة والمنطقة.
- العنوان.
- بيانات مسؤول التواصل.
- بيانات المالك أو المفوض.
- رقم السجل التجاري.
- نوع النشاط.
- وسائل التواصل.
- مناطق الخدمة المقترحة.

### المستندات المطلوبة
المستندات الأساسية المطلوبة للمراجعة:
- ترخيص الشركة أو ترخيص مزاولة النشاط.
- عقد التأسيس.
- بيانات المالك أو المفوض الرسمي.
- السجل التجاري.

ممكن إضافة مستندات حسب الدولة:
- البطاقة المدنية أو الهوية للمالك.
- شهادة ضريبية إن وجدت.
- تفويض رسمي إذا مقدم الطلب ليس المالك.
- شهادة صحية أو ترخيص غذائي حسب البلد.

### قرار الأدمن
الأدمن يراجع:
- اكتمال البيانات.
- صحة المستندات.
- تطابق اسم الشركة/المطعم مع السجل والترخيص.
- صلاحية المستندات.
- أهلية المطعم للعمل داخل المنطقة.

القرار يكون:
- قبول.
- رفض مع سبب.
- طلب استكمال أو إعادة رفع مستند.

## Logic Analysis
### State Machine
الحالات المناسبة لتسجيل المطعم:
- `Draft`: المطعم بدأ الطلب ولم يرسله بعد.
- `PendingDocuments`: البيانات موجودة لكن المستندات ناقصة.
- `SubmittedForReview`: الطلب مكتمل وتم إرساله للأدمن.
- `NeedsChanges`: الأدمن طلب تعديل بيانات أو إعادة رفع مستند.
- `Rejected`: الطلب مرفوض مع سبب.
- `Approved`: الطلب مقبول تجاريًا وقانونيًا.
- `NeedsOperationalSetup`: يحتاج إعداد قوائم/أسعار/طاقة/مناطق قبل التشغيل.
- `ReadyToGoLive`: جاهز للتفعيل.
- `Active`: المطعم ظاهر ويستقبل طلبات.
- `Suspended`: المطعم موقوف مؤقتًا.
- `Terminated`: تم إنهاء التعامل.

### منطق رفع المستندات
- كل مستند له نوع وحالة مستقلة.
- لا يمكن إرسال الطلب للمراجعة إلا بعد رفع المستندات المطلوبة.
- يمكن للأدمن رفض مستند واحد وطلب إعادة رفعه بدون رفض الطلب كاملًا.
- كل نسخة مستند تحفظ كـ version ولا يتم حذف النسخ القديمة.

### منطق المراجعة
- الأدمن لا يعتمد المطعم إذا كان هناك مستند مطلوب ناقص أو مرفوض.
- قرار القبول أو الرفض يحتاج سببًا أو ملاحظة.
- كل قرار يسجل AuditLog.
- المطعم المعتمد لا يصبح Active تلقائيًا إلا بعد إكمال الإعداد التشغيلي.

### الخصوصية
مستندات المطعم تحتوي بيانات حساسة. لذلك:
- المستندات تعرض فقط للأدمن المصرح.
- روابط الملفات تكون signed URLs.
- يتم تسجيل من شاهد أو حمّل المستند عند الحاجة.
- لا تظهر بيانات المالك الحساسة إلا حسب الصلاحية.

## Current Improvement Direction
1. جعل التسجيل متعدد الخطوات: بيانات المطعم، بيانات المالك، المستندات، الإرسال.
2. جعل مراجعة الأدمن إلزامية قبل الاعتماد.
3. فصل `Approved` عن `Active`.
4. ربط المستندات بـ F007 Document Verification.
5. إضافة أسباب رفض منظمة.
6. إضافة Audit كامل لكل قرار ومراجعة مستند.
