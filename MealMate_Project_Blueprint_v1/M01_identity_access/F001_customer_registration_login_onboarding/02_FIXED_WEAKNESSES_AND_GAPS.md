# F001 — Fixed Weaknesses & Gaps

## الهدف
إغلاق فجوات تسجيل العميل بدون إضافة بطء أو مراجعة إدارية.

## القرار النهائي
**تسجيل العميل فوري. لا توجد مراجعة أدمن، ولا Approval، ولا انتظار.**

بعد التسجيل الناجح:
- يتم إنشاء الحساب فورًا.
- يتم إصدار session/token فورًا.
- يدخل العميل التطبيق مباشرة.
- يتم توجيهه إلى اختيار الاشتراك أو استكمال بيانات بسيطة.

## الفجوات التي تم إغلاقها تصميميًا
- فصل onboarding عن auth flows.
- إزالة أي مراجعة أو اعتماد إداري من تسجيل العميل.
- تعريف customer account state machine مناسبة للعميل.
- جعل OTP/Email verification غير معطّل للدخول العام.
- تقييد العمليات الحساسة فقط عند عدم التوثيق.
- تصميم APIs قصيرة ومباشرة.
- إضافة funnel KPIs.
- توثيق منع التكرار وrate limiting.

## قرارات التصحيح المعتمدة
### Business
- لا نضع احتكاكًا أمام العميل الجديد.
- الهدف هو تقليل زمن التسجيل وزيادة first subscription conversion.
- التحقق الأمني يجب ألا يتحول إلى طابور مراجعة.
- أي نقص في التوثيق يظهر كدعوة لاستكمال التحقق، لا كحظر للتطبيق.

### Logic
- `Guest` يتحول إلى `Active` مباشرة بعد التسجيل.
- `VerificationPending` حالة مرافقة للحساب النشط، وليست حالة انتظار.
- لا توجد transitions باسم `Approve` أو `Reject` لتسجيل العميل.
- `Suspended` فقط تمنع الدخول أو الاستخدام، وتأتي من قرار أمني/إداري لاحق.

### Data / API / Security
- التسجيل يعتمد على uniqueness للهاتف أو البريد.
- OTP له rate limit وexpiry.
- التسجيل يستخدم Idempotency-Key.
- الدخول والجلسات تعتمد token/session آمن.
- العمليات الحساسة يمكن أن تتطلب توثيقًا قبل التنفيذ.
- Audit يسجل التسجيل والدخول وفشل الدخول والتوثيق.

## ما الذي لا نفعله
- لا نرسل العميل إلى قائمة مراجعة أدمن.
- لا ننتظر موافقة لتفعيل الحساب.
- لا نمنع فتح التطبيق بسبب عدم توثيق OTP.
- لا نستخدم endpoint عام طويل مثل `customer_registration_login_onboarding` للتسجيل الفعلي.

## حالة المعالجة
الفجوة الخاصة بالبطء والمراجعة تم إغلاقها. التصميم الحالي يعتمد **Instant Customer Registration**.
