# F003 — Feature Analysis

## Context
الميزة ضمن **M01 — إدارة الحسابات والوصول** وتؤثر على نطاق الموديول ووظائفه الأساسية.

## Business Analysis
### 1.1 عدم تحديد نموذج علاقة السائق
التوثيق لا يوضح هل السائق:
- تابع لمطعم.
- تابع للمنصة.
- يعمل في منطقة محددة.
- يمكنه العمل مع أكثر من مطعم.

الأثر: صعوبة بناء قواعد الإسناد والتسعير والمحاسبة لاحقًا في موديولات التوصيل والتشغيل.

### 1.2 عدم تحديد معايير أهلية السائق
لا توجد معايير واضحة للقبول:
- رخصة قيادة.
- هوية.
- وسيلة نقل.
- رقم هاتف موثق.
- منطقة عمل.
- حد أدنى لعمر أو نوع رخصة حسب الدولة.

الأثر: قبول سائقين غير مؤهلين أو رفض سائقين بدون سبب موحد.

### 1.3 لا يوجد SLA لاعتماد السائق
غياب زمن مراجعة واضح يؤدي إلى بطء التوسع التشغيلي.

الأثر: نقص في السائقين وقت زيادة الطلب أو فتح مناطق جديدة.

### 1.4 عدم ربط الاعتماد بمخاطر خدمة العملاء
السائق يتعامل مع بيانات العميل وموقعه والتسليم. التوثيق لا يربط التسجيل بسياسات الخصوصية والاتصال الآمن.

الأثر: مخاطر خصوصية وتشغيل في مرحلة التوصيل.

## Logic Analysis
### 2.1 حالات السائق غير كافية
السائق يحتاج حالات تشغيلية مختلفة عن المستخدم العادي:
- `PendingDocuments`
- `PendingReview`
- `Approved`
- `Active`
- `Offline`
- `Suspended`
- `ExpiredDocuments`
- `Blocked`

الأثر: قد يبقى السائق Active رغم انتهاء وثيقة أو تعليق إداري.

### 2.2 لا يوجد منطق لانتهاء الوثائق
وثائق السائق قد تنتهي صلاحيتها. التوثيق لا يحدد:
- تاريخ انتهاء كل وثيقة.
- إشعار قبل الانتهاء.
- تعليق تلقائي عند الانتهاء.
- إعادة التحقق بعد الرفع.

الأثر: مخاطر قانونية وتشغيلية.

### 2.3 عدم تعريف نطاق العمل
لا يوجد منطق واضح لـ:
- country scope.
- region scope.
- restaurant assignment.
- maximum service radius.

الأثر: سائق قد يستلم طلبات خارج منطقته أو خارج المطعم المرتبط به.

### 2.4 لا يوجد منطق إعادة التفعيل
إذا تم تعليق السائق، لا توجد خطوات واضحة للعودة:
- من يراجع؟
- ما الوثائق المطلوبة؟
- هل التقييمات السابقة تؤثر؟

الأثر: قرارات تشغيلية غير موحدة.

## Data / API / Security Analysis
### 3.1 نموذج بيانات السائق ناقص
ينقص:
- `DriverProfile`
- `VehicleInfo`
- `DriverDocument`
- `DriverRegion`
- `DriverApprovalDecision`
- `DriverStatusHistory`

### 3.2 APIs عامة
المطلوب endpoints متخصصة:
- submit driver registration.
- upload identity/license.
- approve/reject.
- suspend/reactivate.
- update service region.

### 3.3 الخصوصية غير مفصلة
ينقص تحديد ما يراه السائق من بيانات العميل:
- الاسم؟
- الهاتف؟
- العنوان؟
- موقع حي أم دقيق؟
- متى يظهر زر الاتصال؟

## Current Improvement Direction
## Driver Profile Photo Requirement
الصورة الشخصية للمندوب جزء أساسي من ميزة تسجيل واعتماد المندوب، وليست اختيارية مثل صورة العميل. الهدف منها دعم مراجعة الأدمن، وتقليل انتحال الهوية، وتسهيل التعرف على المندوب داخليا أثناء التشغيل والدعم وتسليم الطلبات.

### Business View
- يجب رفع صورة شخصية واضحة للمندوب قبل إرسال طلب الاعتماد أو قبل قرار الاعتماد النهائي.
- الأدمن يراجع الصورة مع الهوية ورخصة القيادة وبيانات وسيلة النقل.
- لا يتم تفعيل المندوب كـ `Active` إذا كانت الصورة ناقصة أو غير واضحة أو لا تطابق بيانات الهوية.
- يمكن للعميل رؤية صورة المندوب أثناء التوصيل فقط إذا كان ذلك منصوصا عليه في سياسة الخصوصية وتجربة المنتج، ويفضل عرض الاسم الأول والصورة فقط بدون بيانات حساسة.

### Logic View
- الحالة `PendingDocuments` تظل مستمرة إذا كانت الصورة الشخصية غير مرفوعة.
- الحالة `SubmittedForReview` لا تقبل الطلب إلا بعد اكتمال البيانات الأساسية والمستندات والصورة الشخصية.
- إذا رفض الأدمن الصورة فقط، يتحول الطلب إلى `NeedsChanges` مع سبب واضح مثل: صورة غير واضحة، وجه غير ظاهر، ملف غير صالح، أو صورة لا تخص المندوب.
- تحديث الصورة بعد تفعيل الحساب يجب أن ينشئ طلب مراجعة أو حالة `ProfileImageReviewRequired` حسب سياسة التشغيل.

### Data / API / Security View
- إضافة الحقول:
  - `ProfileImageUrl`
  - `ProfileImageStorageKey`
  - `ProfileImageStatus`
  - `ProfileImageReviewedBy`
  - `ProfileImageReviewedAtUtc`
  - `ProfileImageRejectionReason`
- إضافة endpoint متخصص:
  - `POST /api/v1/drivers/registrations/{id}/profile-image`
- قيود الرفع:
  - صيغ مسموحة: JPG, PNG, WebP.
  - حجم أقصى محدد من المنتج/التقنية.
  - فحص نوع الملف الحقيقي وليس الامتداد فقط.
  - تخزين آمن مع signed URL أو access-controlled media.
  - تسجيل Audit عند الرفع أو التغيير أو الرفض.

1. **تعريف State Machine**
   - تحديد كل حالة مسموحة.
   - تحديد الانتقالات المسموحة والممنوعة.
   - ربط كل انتقال بصلاحية وسبب وAudit Log.

2. **تفصيل API Contract**
   - إضافة schemas لكل endpoint:
     - `GET /api/v1/driver_registration_approval`
     - `GET /api/v1/driver_registration_approval/{id}`
     - `POST /api/v1/driver_registration_approval`
     - `PUT /api/v1/driver_registration_approval/{id}`
     - `POST /api/v1/driver_registration_approval/{id}/actions/{action}`
   - توثيق status codes: `200`, `201`, `400`, `401`, `403`, `404`, `409`, `422`, `500`.
   - اعتماد `ProblemDetails` للأخطاء و`CorrelationId` للتتبع.

3. **تفصيل صلاحيات Role/Scope**
   - ربط كل شاشة وكل endpoint بصلاحية مثل `F003.View`, `F003.Create`, `F003.Update`, `F003.Override`.
   - تحديد Scope حسب الدولة والمنطقة والكيان المرتبط.

4. **تعريف نموذج البيانات**
   - تحديد الجداول/الـ collections.
   - تحديد العلاقات، الفهارس، unique constraints، وretention policy.
   - إضافة `RowVersion` أو optimistic concurrency.
