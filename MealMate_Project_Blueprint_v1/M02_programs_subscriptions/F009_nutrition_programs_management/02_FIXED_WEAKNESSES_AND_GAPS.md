# F009 — Fixed Weaknesses & Gaps

## الهدف
هذا الملف يحول نقاط الضعف والفجوات إلى قرارات تصميمية مغلقة قابلة للتنفيذ.

## الفجوات التي تم إغلاقها تصميميًا
1. كتابة state machine مخصصة للميزة.
2. تعريف business owner وoperational owner.
3. تفصيل API contract كامل.
4. تفصيل data model والقيود والفهارس.
5. ربط كل عملية بـ RBAC + Scope + Audit.
6. إضافة KPIs خاصة بالميزة.
7. تحويل معايير القبول إلى test scenarios.
8. تحديد أثر الميزة على الموديولات المرتبطة.
9. تثبيت السعر وقت الشراء داخل `SubscriptionSnapshot`.
10. تعريف lifecycle واضح للاشتراك: `Draft`, `PendingPayment`, `Active`, `Upgraded`, `Renewed`, `Cancelled`, `Expired`.
11. دعم idempotency في الترقية والتجديد ومنع التحصيل أو التحديث مرتين.
12. ربط كل قرار تسعير بـ `PricingRuleVersion` حتى يكون قابلا للتفسير.
13. تثبيت محتوى الباقة وقت الاشتراك، بما يشمل عدد وقائمة المطاعم المتاحة.

## قرارات التصحيح المعتمدة
### Business
- كل ميزة يجب أن تملك KPI واحدًا على الأقل لقياس النجاح.
- كل قرار يدوي يحتاج owner وSLA وسبب واضح.
- أي أثر على العميل أو المطعم أو المالية يجب أن يظهر في الواجهة أو التقرير المناسب.
- الاشتراكات القديمة لا تتأثر بتغيير الأسعار لاحقا، لأن السعر والخصومات والعملة تحفظ وقت الشراء.
- أي تغيير في الأسعار أو قواعد الخصم يطبق على الاشتراكات الجديدة فقط إلا لو هناك قرار مالي واضح ومراجع.
- الاشتراكات القديمة لا تحصل تلقائيا على مطاعم تمت إضافتها للباقة بعد الشراء، لأن محتوى الباقة محفوظ في `SubscriptionSnapshot`.

### Logic
- الحالات يجب أن تكون explicit ولا يسمح بالانتقال بينها إلا عبر action محدد.
- كل action حساس يحتاج idempotency وconcurrency control.
- أي override يجب أن يكون مسببًا ومقيدًا بصلاحية ونطاق.
- الخصم يطبق مرة واحدة وبترتيب واضح مع السعر الأساسي والتصنيف والعملة.
- الترقية والتجديد يتطلبان `Idempotency-Key` لمنع تكرار التحصيل أو تحديث الاشتراك مرتين.
- لا يتم إعادة حساب اشتراك قديم باستخدام قواعد التسعير الحالية؛ يتم الرجوع إلى الـ snapshot المحفوظة.
- لا يتم عرض مطاعم الاشتراك القديم من تعريف الباقة الحالي مباشرة، بل من قائمة المطاعم المحفوظة في snapshot وقت الاشتراك.

### Data / API / Security
- كل API يرجع ProblemDetails عند الخطأ.
- كل entity مهم يحتوي Status وCreatedAtUtc وUpdatedAtUtc وRowVersion.
- كل عملية حساسة تسجل AuditLog مع before/after/reason/correlationId.
- كل صلاحية تطبق Permission + Scope وليس Permission فقط.
- `SubscriptionSnapshot` يجب أن يحتوي السعر الأساسي، الخصومات، التصنيف، العملة، مدة الاشتراك، ونسخة قواعد التسعير.
- `PricingRuleVersion` يجب أن يحفظ مع كل عملية شراء أو ترقية أو تجديد.
- `SubscriptionSnapshot` يجب أن يحتوي `RestaurantCount`, `IncludedRestaurantIds`, و`PackageContentVersion`.

## حالة المعالجة
Design gaps are considered **closed for implementation planning**. أي تغيير لاحق يجب أن ينعكس في هذا الملف وملف spec.
